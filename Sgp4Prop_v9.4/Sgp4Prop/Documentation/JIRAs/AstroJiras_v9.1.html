# General

Â 

This package contains examples of using the **Astrodynamics Standards (AS)** libraries in Swift.

Please read the Programmer's Guide for notes regarding the Mac.

Â 

### Friction

Â 

Because calling C functions is common, the Swift compiler can read and interpret C header `(*.h)` files.Â  Among other things, C function prototypes are converted to Swift equivalents; that conversion is exact but, because of the friction between them (C and Swift are different in not just their syntax but also their style and concepts) that conversion can be convoluted and unnatural.

Â 

Each Swift function is of the form:

Â 

```

Â Â Â  public func OpenLogFile(_ fileName: UnsafeMutablePointer<CChar>) -> Int32 {

Â Â  Â Â Â Â Â let function = unsafeBitCast(getFunctionPointer(libHandle, "OpenLogFile"), to: fnPtrOpenLogFile.self)

Â Â Â Â Â Â Â  return function(fileName)

Â Â Â  }

```

Leaving aside the syntax differences, Swift's version of that same function has one parameter, a pointer to an array of `CChars` (8-bit signed integers) and returns an Int32 (a 32-bit signed integer); the same as the C function, of course.

Â 

The `UnsafeMutablePointer` type in Swift indicates (`Unsafe`) meaning Swift has had to give up some control over the management of this parameter, it does **not** mean "unsafe" in the sense that a C pointer is dangerous when changed.Â  The `Mutable` part says the called function could change the pointee (in this case, the character array).Â  [Note, however, in this case, that array is the name of a file to be opened and we can be pretty sure `OpenLogFile` is not going to change it, so `Mutable` is surprising. But because the C function parameter has no `const` ornamentation, Swift must assume mutability.]Â  The `Pointer` part of the type name is because it is a pointer.

Â 

The Swift inside the wrapper is getting the function pointer from the library (essentially calling `dlsym`), attributing to it the function's signature (from the C header file) and calling the function.

Â 

## Swift differences

Â 

### Strings:

Â 

- Swift strings are not byte-by-byte character arrays because, in general, they are Unicode strings and Unicode characters may be multiple bytes long.Â  However, in the limited case of strings of only ASCII characters, the text of the string will happen to be stored as contiguous array of signed bytes.

Â 

- Swift strings are not NULL terminated, as they are in C, so conversions are necessary when passing strings back and forth.

Â 

- Conventionally, when the purpose of the Swift function is to return a string, it is delivered as the return value of the function and not via a pointer parameter.

Â 

### Integers

Â 

- In C `int` is 32-bits wide; in Swift `Int` is 64-bits wide.Â  C is using an unnatural `__int64` to assure 64-bits and Swift has to use the somewhat unnatural `Int32` to match C's `int`.

Â 

### Arrays

Â 

- Swift's array support is primitive, especially multi-dimensional arrays, and even more so when being passed into or out of functions.

Â 

### Pointers

Â 

- Pointers have been mentioned above and Swift is strict about pointer types. In C, the type that pointer is aimed at is specified (eg `double*`) but that can be coerced (explicitly or implicitly) to any other pointer type. If you try that in Swift it will not compile, the pointee of a pointer is part of the type; `UnsafePointer<Int>` and `UnsafePointer<Double>` are forever different.

Â 

### Optionals

Â 

- The Optional type is not explicitly available in C, and its implicit use is sometime accidental.Â  In Swift, an `Optional` represents the strongly typed absence of a value.Â  An Optional string and an Optional number may both be NULL under the covers but they retain their string and number qualities.

Â 

Â  Optionals are often used as the return value of a function when the function could fail.Â  The return value of, for example, `Integer("string")` is an optional number because `"string"` might be `"124789"` or it might be `"wombat"`.Â  In the first case the returned optional number can be resolved to a number, in the case of the Australian marsupial it cannot.Â  This `Integer` function can consume any string, including nonsense, but it will always return a value and it will not fail.

Â 

### Functions

Â 

- Swift's function parameters are usually call-by-value.Â  Though catered for in the language, call-by-reference usage is less common.Â  Values generated inside a function are typically returned as the function value (if multiple values are returned, that's usually contained in a `struct`).

Â 

## Why do we care?

Â 

The Swift wrapper functions are all exact analogs of the equivalent C functions, but several of them can be expressed better by taking advantage of Swiftâs differences.

Â 

Take the use of strings, for example.Â  The library function `DllMainGetInfo` is defined in the C style as:

Â 

```

Â Â Â  void DllMainGetInfo(char infoStr[128]);

```

The direct equivalent in Swift is:

```

Â Â Â  DllMainGetInfo(_ infoStr: UnsafeMutablePointer<CChar>)

```

In this direct C equivalent, the programmer has to deal with string conversion by writing repetitive preamble code which reduces readability and increases the cross-section for errors.Â  There are several library functions which have the sole purpose of obtaining a string result.

Â 

The idiomatic Swift version would be:

```

Â Â Â Â  dllMainGetInfo() -> String

```

with string conversion inside the function, and in cases where the string emitting function could fail, an optional string would be returned.

Â 

A lesser friction appears when an error code is emitted as a C `int`, a 32-bit value (typically zero or non-zero) and the Swift programmer's natural integer is 64-bits, forcing a coercion to avoid compiler complaints.

Â 

### Driver on wrappers

Â 

In those cases where a more 'Swifty' equivalent function could be used they are provided in the `drivers/Drivers.swift` file, and the programmer can make a choice.Â  Note that many of the library functions have no Swifty equivalent, because having one wouldnât improve how it would be called from Swift.

Â 

Essentially, these driver functions are a thin layer of code laid over the thin layer that performs the basic Swift matching to the dynamic library entry points.

Â 

The 'Swifty' functions must be named differently from their equivalents.

Hey Tom,

Yes, the astrostandards libraries have to be code-signed in order to use them as part of a distributable application. I do this in the last step in my Astrostandards post-processing script (see below). Also, I remove the 'quarantine' bit which can prevent a library from being executed at runtime (see xattr command below). This bit can be set by the OS when a file or library is downloaded from the internet.

John

# Codesign astrostandards dylibs in the destination astrostandards directory or else
