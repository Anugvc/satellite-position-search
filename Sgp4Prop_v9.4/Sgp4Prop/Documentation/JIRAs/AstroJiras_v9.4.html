ASTROSTDS-1410 Done 9.4 Bug

Logic error in JNA TwoToOneDim method in the Utility class

Got email from David Driggs (Dave.Driggs@ngc.com) on 4/11/2024:

Dear Sirs:

I discovered a bug in the AstroStds 9.3 Java sample code. The TwoToOneDim method in the Utility class has a logic error in the array bound checking. If the input array bounds match the specified rows and columns it throws an exception. I believe the logic should be reversed - if the array bounds do *not* match, throw and exception. See line 284 in SampleCode/Java/jna/src/afspc/astrostds/utilities/Utility.java

Change from:

        if (source.length == rows && source[0].length == columns)

to:

        if (!(source.length == rows && source[0].length == columns)) Regards, Dave

ASTROSTDS-1353 Done 9.4 Improvement

Design architecture to fix write permission issue in Jenkins

The problem is that we have directories being created by Windows, Linux, and Mac.   And the Administrator/root user is not the same on any of those OS’s.   So, directories created by Windows may not have permission to be written to by Linux root or Mac root user, and vice-versa.   And there’s no way of knowing which OS creates the directories first.   The Jenkins manager controls that.   So, when you build all, several of the pipelines would fail because it write permission issues.

ASTROSTDS-1332 Done 9.4 Improvement

Investigate Matrix inversion methods

Felix has better convergence than Astro Standards.  One possible cause could be the matrix inversion.  AS uses Gauss-Jordan while he uses Cholesky.  There are a few methods coded up to investigate:

Gauss-Jordan (already in AS)

Cholesky

Root-free Cholesky

Givens (slow)

Householder

 

*Update:* Investigated Cholesky and Root-free Cholesky and got the same answers as Gauss-Jordan.  However, it seems like Root Free Cholesky could be faster but slightly less accurate.  Givens is probably too slow to worry about testing though we might be able to swap row/column to make it faster.  It still isn't as accurate as Householder.
