ASTROSTDS-1301 Done Task

Update Sensor document to remove minimum elevation parameter from the constant azimuth fan

Since the fan can be tilted, the minimum elevation can be inadvertently applied and produce wrong results. For example, when the tilt angle is 90 degrees, the fan becomes a local horizontal plane and therefore the user has to set minimum elevation to something less than zero degrees for LAMOD to return any passes. Obviously that's something that the user can easily forget. To avoid this kind of problem, removing minimum elevation from constant azimuth fan seems to be a better solution 9.3

ASTROSTDS-1283 Done Bug

Fix round-off error when over punching different observation data

Summary: If a negative declination value with more than 4 decimal places is in the Astro Standards memory, then an error can occur when creating the B3 observation string for that ob. (This is theoretically possible for elevation and range rate values as well.) The specific example we observed was a type 5 observation with a declination value of -29.99997 which resulted in a declination value of K00000 (-20.0000) in the B3 observation. Based on this one example, it appears that the negative number to letter conversion is being done before rounding to 4 decimal places resulting in an error of approximately 10 degrees.

 

Disclaimer: We are using the Astro standards for modeling and simulation purposes and are very likely doing things that may not be likely to occur with typical usage.

 

Astro Standards Version: 8.4

 

Steps to reproduce the error:
# Add a type 5 observation to the Astro Standards memory
# Change the declination value to -29.99997
# Get the B3 string for that ob from the Astro Standards and note the declination value
